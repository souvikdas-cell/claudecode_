# StrikX Project LLM — Last 1 Month Learning Report

## 1. Executive Summary

This report covers the last 30 days of Project LLM interactions around StrikX / StrikeX / Trader Mode. The scope is based on the user’s instruction to focus only on last-1-month conversations, not the full historical memory. The uploaded prompt explicitly defines last-1-month chats as the primary source of truth and asks not to let old project memory dominate the report. 

The last month was not about building one clean PRD. It was about converting scattered StrikeX work into usable product, campaign, UI, analytics, and LLM-training assets.

The strongest learning is this:

**StrikeX is no longer being treated as a standalone feature name. It is being shaped into Trader Mode, a clearer F&O workspace inside FinX.**

The main work in the last month happened across seven clusters:

1. AMJ operating clarity for StrikeX.
2. Trader Mode positioning and naming.
3. Plotline-safe HTML widgets and walkthroughs.
4. Weekly Analysis strategy and card redesign.
5. Pick Your View strategy-entry campaign.
6. Web redirection and Mixpanel/MoEngage event planning.
7. Prompt-building and Project LLM training workflows.

The biggest correction was around design and delivery quality. Static visuals, unreadable mobile cards, clipped layouts, broken CTA behavior, wrong naming, expired Sensex/Nifty dates, and overdesigned outputs were repeatedly rejected and corrected.

The Project LLM learned that this project needs high constraint-following. It must preserve exact dimensions, exact flows, exact CTA behavior, exact brand rules, and exact data logic. Generic PM summaries are not enough.

## 4. Major Learning Themes

### Product learnings

1. StrikeX is being reframed as Trader Mode.
2. Trader Mode works better as a workspace label.
3. StrikeX still matters as backend or legacy recognition.
4. Web adoption is a major priority.
5. Strategy Builder has stronger pull.
6. Strategy Finder helps reduce first-time confusion.
7. Price Ladder and Multiple Charts need guided education.
8. Weekly Analysis should not only show losses.
9. Weekly Analysis should explain what went wrong and route users to Trader Mode.
10. Product education is now part of the product, not only campaign copy.

### Prompt-engineering learnings

1. Generic prompts fail for this project.
2. Prompts must include screen sequence, dimension, brand tokens, motion behavior, CTA behavior, and output format.
3. Claude prompts need explicit code-generation rules.
4. GPT image prompts need hard constraints to avoid collage outputs.
5. Prompt-building should first ask whether output is for Claude or GPT.
6. When creating prompts for HTML, the model must include Plotline/WebView constraints.
7. Reusable prompts should include “no extra explanation” when user wants direct output.

### LLM-training learnings

1. Project LLM must learn from corrections, not only final answers.
2. User rejection language is valuable training data.
3. Repeated corrections reveal durable rules.
4. LLM must not add unnecessary columns, suggestions, or design variants when user asks for narrow edits.
5. Project memory should be updated only after durable decisions.
6. Last-1-month chat learning should be separated from full project memory.

### Data and analytics learnings

1. Event tracking must be clean and minimal.
2. Only filled columns should be added to tracking sheets.
3. Redirection analytics must be summarized for executives, not dumped.
4. Dynamic action IDs and source fields matter for campaign attribution.
5. Expiry-based URLs must be checked before sharing.
6. Static expired values create product credibility risk.
7. “Not set” or incomplete tracking remains a risk from memory-only context, not verified in all last-month chats.

### UX/UI learnings

1. Mobile readability is the top constraint for Weekly Analysis.
2. 870×555 cards cannot carry dense information.
3. Old large-font cards were more readable than new dense cards.
4. Horizontal card flow must preserve each card as standalone.
5. HTML widgets need deterministic layout.
6. iOS behavior must be tested separately from Android.
7. Animation must not overlap text.
8. Mouse hover animations improve walkthrough comprehension.
9. Screen transitions need delay so users can read instructions.
10. Text color must be high contrast against white UI screenshots.

### Business logic learnings

1. Nifty and Sensex expiries cannot be hardcoded loosely.
2. Gold and Crude need separate underlying and expiry handling.
3. Strategy links must load based on correct underlying, expiry, and strategy key.
4. “Pick Your View” is a decision-entry concept, not only a campaign banner.
5. Weekly Analysis must connect trader losses to usable tools.
6. A VP Research YouTube walkthrough can be used for dropped-off users, but success depends on CTA placement inside Trader Mode.

### Workflow/process learnings

1. Canvas should be default for HTML/UI builds.
2. User prefers edits over redesign when in editing mode.
3. User rejects unnecessary reinvention.
4. For executive comms, shorten and sharpen.
5. For tracking sheets, do not add explanatory columns.
6. For images, generate one standalone asset at a time.
7. For HTML, test CTA, clipping, animation, scroll, and date logic before declaring completion.

## 6. Prompt Evolution

### Earlier prompt weaknesses

1. Prompts were often outcome-led but not constraint-led.
2. HTML prompts did not always hardcode Plotline restrictions.
3. Image prompts allowed models to create collages or multiple designs in one canvas.
4. UI prompts lacked enough hard rules for mobile readability.
5. Walkthrough prompts did not always define screen timing, hover behavior, and text instructions.
6. Analytics prompts allowed extra columns and unwanted explanatory fields.

### Better prompt structures discovered

1. Start with target model: Claude or GPT.
2. State exact role.
3. State exact artifact.
4. State exact dimensions.
5. State exact flow.
6. State exact brand tokens.
7. State exact animation behavior.
8. State exact CTA behavior.
9. State hard exclusions.
10. State output format.

### Reusable prompt patterns

#### For Claude HTML

Use when generating Plotline-safe HTML.

```text
You are building a single self-contained HTML file for Plotline/WebView.
Use Onest font.
Use Choice colors.
No DOCTYPE.
No inline handlers.
Use addEventListener only.
No external assets unless already embedded.
Keep layout mobile-safe.
Preserve exact dimensions.
Return only final HTML code.
```

#### For walkthrough

```text
Build a screen-by-screen walkthrough.
Use animated mouse hover and click.
Show instruction text on every screen.
Add delay between screens.
Do not zoom in a way that clips UI.
Keep text visible over white screenshots.
Use #2777F3 for instruction highlights.
```

#### For image generation

```text
Do not create a collage.
Generate one standalone design.
No labels like Variant A or Design B.
No extra white space outside the banner.
Same dimensions for every output.
```

#### For analytics sheet

```text
Only add the columns that are already filled.
Do not add extra explanatory columns.
Preserve the existing sheet structure.
```

### Failed prompt patterns

1. “Make it better” without dimensions.
2. “Use all screenshots” without hierarchy.
3. “Create multiple designs” without anti-collage rules.
4. “Make mobile-friendly” without font-size constraints.
5. “Create walkthrough” without mouse animation and screen delay.
6. “Add event tracking” without sheet-column limits.

### Final recommended prompt style for this project

```text
Target model: Claude / GPT

Task:
Create [artifact] for [surface].

Context:
[Product context]
[User segment]
[Business objective]

Hard constraints:
1. [Dimension]
2. [Brand]
3. [Platform]
4. [CTA behavior]
5. [No extra columns / no collage / no redesign / etc.]

Exact flow:
Screen 1:
Screen 2:
Screen 3:

Output:
Return only [final HTML / final prompt / final table].
Do not add explanation.
Do not change the structure.
```

## 8. Mistakes, Corrections, and Iterations

### 1. Mistake: Generic memory summary

**What happened:** First report risk was summarizing full project memory.
**Correction:** User clarified that the report must use last-1-month chats only.
**What changed:** This report separates recent chat evidence from memory-only context.
**Should not repeat:** Do not default to full project memory when user asks for last-month learning.

### 2. Mistake: Overdesigned Weekly Analysis cards

**What happened:** Cards became unreadable on mobile.
**Correction:** User asked to cut short content, remove 70% if needed, and use older visible font sizes.
**What changed:** Mobile readability became the top design constraint.
**Should not repeat:** Do not compress many insights into 870×555 cards.

### 3. Mistake: Static walkthrough flow

**What happened:** Walkthrough looked static and flowless.
**Correction:** User asked for mouse hover, click animation, instruction text, and screen delays.
**What changed:** Walkthrough became a guided product education asset.
**Should not repeat:** Do not show screenshots without explanatory motion.

### 4. Mistake: Screen zoom causing clipping

**What happened:** Screen 1 and 2 were zoomed and clipped.
**Correction:** User asked not to zoom in that way.
**What changed:** Full-screen context had to be preserved.
**Should not repeat:** Avoid zoom effects that break product understanding.

### 5. Mistake: CTA animation broke after layout got fixed

**What happened:** Visual layout improved but CTA animation ruptured.
**Correction:** User asked to cross-check after changes.
**What changed:** QA must include animation behavior, not just static layout.
**Should not repeat:** Do not validate only the screenshot state.

### 6. Mistake: Expired Sensex expiry shown

**What happened:** 30 April expiry still appeared after date passed.
**Correction:** User flagged date logic issue.
**What changed:** Expiry logic must be dynamic and time-aware.
**Should not repeat:** Do not use stale expiry dates in trading widgets.

### 7. Mistake: Broken Gold/Crude URLs

**What happened:** Generated URLs did not work.
**Correction:** User asked for reference Nifty URL to share with developers.
**What changed:** URL structure should be validated against existing working reference.
**Should not repeat:** Do not invent deep-link patterns without reference validation.

### 8. Mistake: Extra information in tracking sheet

**What happened:** Assistant added unnecessary tracking-plan columns.
**Correction:** User said only add filled columns.
**What changed:** Analytics sheet output must preserve exact input structure.
**Should not repeat:** Do not enrich tracking plans unless asked.

### 9. Mistake: Collage-style image outputs

**What happened:** GPT generated multiple designs in one image despite instructions.
**Correction:** User required hardcoded anti-collage prompt rules.
**What changed:** Image prompts now need explicit one-output-at-a-time constraints.
**Should not repeat:** Do not ask for multiple variants in one canvas unless user explicitly wants a board.

### 10. Mistake: Assuming auto-scroll works across platforms

**What happened:** Auto-scroll worked on Android but not iOS 16.
**Correction:** User rejected bottom sheet and asked for scroll affordance inside tabs.
**What changed:** iOS-safe design needs visible partial pills or affordance.
**Should not repeat:** Do not rely on platform behavior without iOS validation.

In the last 30 days, the StrikX / StrikeX project moved from broad product memory into execution-heavy learning. The main product shift was toward **Trader Mode** as a clearer workspace framing for F&O users. Strategy Builder remained the strongest pull lever. Strategy Finder remained useful for reducing first-time confusion. Pick Your View evolved as a strategy-entry campaign, but dynamic expiry and deep-link correctness became critical. Weekly Analysis shifted from a dense, guilt-heavy loss report to a cleaner risk-leak narrative that should sympathise, diagnose, and redirect users to Trader Mode. The 7-card PNG infra remained fixed, so design had to adapt within that system. Mobile readability became a hard rule. Old large-font cards became the benchmark. Plotline HTML work improved through repeated corrections around CTA behavior, iOS issues, animation overlap, clipping, hover states, screen delay, and visible instructional text. Canvas-first became the default workflow for HTML/UI builds. Prompting matured into model-specific execution. Every prompt-building flow should first identify Claude or GPT. Claude prompts need strict HTML and output constraints. GPT image prompts need hard anti-collage rules. Analytics work clarified that event sheets must stay clean and only include filled columns. Executive summaries must be short, data-led, and VP/SVP-ready. The strongest LLM training value comes from corrections: do not overdesign, do not add unasked columns, do not rely on old memory, do not ignore iOS behavior, do not use stale expiries, do not create static walkthroughs, and do not summarize generically when the user asks for last-month learning.
